## 1. Feature Summary

### Feature A: HIL Feedback Loop with Feedback Router

Users can provide textual feedback instead of only editing/confirming the plan. The system uses a **Feedback Router** to dynamically choose between two revision strategies:

- **Tweak** (surgical edit): `plan_tweak` node makes a targeted edit to the existing draft, preserving the current plan structure. Single LLM call, no reviewer loop. Returns directly to `plan_output` for the user to review.
- **Rewrite** (clean slate): Clears all adversarial state. The `drafter` starts fresh with the user's feedback injected as the primary variable via `revision_notes`. Full adversarial loop (drafter → reviewers → consensus) runs before returning to `plan_output`.

**Key design constraint — Single Summary, No Accumulation**:
A single `hil_summary` field stores a compressed summary of all prior feedback rounds. Each iteration overwrites (not appends) this field. This prevents context bloat from multi-round feedback while preserving essential history for the LLM.

### Feature B: Plan Export
A Markdown download button in `PlanReview`. Pure frontend, zero backend changes.

### Feature C: Interactive DAG View
Replace the custom SVG `GraphDAGView` with React Flow (`@xyflow/react`). Pan, zoom, auto-layout via dagre. Node Trail view unchanged.

## 2. Graph Topology Change

### Before (current v0.3.0)
```
... → consensus_check → plan_output → END
                 ↑              │
           adv_rewrite ←────────┘ (consensus=False)
```

### After (with HIL feedback loop)
```
... → consensus_check → plan_output ──→ END                    (user confirms)
                 ↑              │
           adv_rewrite ←──┐    └──→ feedback_router            (user gives feedback)
                          │           ├─ "tweak" → plan_tweak → plan_output  (loop)
                          │           └─ "rewrite" → drafter → reviewers → ...
                          │                           (fresh start, old state cleared)
                          └── (consensus=False, from reviewers)
```

**Data flow for feedback**:
```
User submits feedback
       ↓
plan_output_node receives dict {"action":"feedback","text":"..."}
       ↓ sets hil_action="feedback", hil_feedback=text
       ↓
feedback_router classifies → "tweak" or "rewrite"
       ↓
  ┌─── tweak ──────────────────────┐
  │ plan_tweak: single LLM call   │
  │ edits draft in-place          │
  │ updates hil_summary           │
  │ → back to plan_output         │
  │   (re-interrupts for review)  │
  └────────────────────────────────┘
  ┌─── rewrite ────────────────────┐
  │ feedback_router clears state: │
  │   adv_round=0, draft="",      │
  │   verdicts cleared            │
  │ sets revision_notes=feedback  │
  │ updates hil_summary           │
  │ → drafter (fresh start)       │
  │ → full adversarial loop       │
  │ → plan_output (re-interrupt)  │
  └────────────────────────────────┘
```

**Node count**: 17 → 19 (add `feedback_router`, `plan_tweak`)

## 3. State Schema Changes

### New fields in TutorState (`src/graph/state.py`)

Add after `revision_notes`:

```python
    # ── HIL feedback loop ────────────────────────────────────────────
    hil_action: str             # "confirm" or "feedback" — set by plan_output
    hil_feedback: str           # User's raw feedback text — set by plan_output
    hil_summary: str            # Compressed summary of ALL prior feedback (single field, overwritten each round)
    feedback_route: str         # "tweak" or "rewrite" — set by feedback_router
```

**Why `hil_summary` is a single overwritten field (not a list)**:
- Prevents context growth across multi-round feedback
- Each round, the LLM compresses `old_summary + new_feedback` into a fresh summary
- Drafter/tweak nodes receive ONE concise context string, not N rounds of raw text
- Keeps token usage bounded regardless of feedback rounds

Total TutorState fields: 22 → 26.

## 4. SSE Protocol — No New Event Types Needed

The existing SSE protocol fully supports the feedback loop:

1. User submits feedback → frontend calls `/resume` with `feedback` field
2. Backend constructs `Command(resume={"action":"feedback","text":"..."})` and resumes graph
3. Graph runs `plan_output` → `feedback_router` → `tweak`/`rewrite` path → `plan_output`
4. At `plan_output`, `interrupt()` fires again → backend emits `{"type":"interrupt","draft":"..."}` SSE event
5. Frontend receives the interrupt, shows `PlanReview` with the updated draft
6. User can give more feedback (loops) or confirm (exits)

**No new SSE event types needed.** The existing `interrupt`, `node_event`, `token`, `text`, `done` events handle everything.

```python
@traced_node
async def plan_output_node(state: TutorState) -> dict:
    """Final plan output — interrupt for HIL review."""
    plan_text = state.get("draft", "")

    try:
        user_response = interrupt(plan_text)
    except ValueError:
        logger.warning("interrupt() failed (no checkpointer?), skipping HIL review")
        user_response = plan_text

    # ── User provided feedback (dict) → route to feedback_router ──
    if isinstance(user_response, dict) and user_response.get("action") == "feedback":
        return {
            "hil_action": "feedback",
            "hil_feedback": user_response.get("text", ""),
        }

    # ── User confirmed (string) → finalize ──
    final_plan = user_response if isinstance(user_response, str) and user_response else plan_text
    return {
        "plan": final_plan,
        "messages": [AIMessage(content=final_plan)],
        "hil_action": "confirm",
    }
```

**Key behavior**:
- `interrupt()` returns whatever `Command(resume=X)` passed as `X`
- If `X` is a dict with `action="feedback"` → set `hil_action="feedback"` and `hil_feedback`
- If `X` is a string → treat as confirmed plan text (backward compatible with current frontend)

**Acceptance**: `plan_output_node` returns `hil_action="feedback"` for dict input, `hil_action="confirm"` for string input.

#### Node: `feedback_router` — Classify feedback as tweak or rewrite

Uses the supervisor's Qwen model (fast, cheap) with structured output.

```python
@traced_node
async def feedback_router(state: TutorState) -> dict[str, Any]:
    """Classify user's HIL feedback as 'tweak' (minor edit) or 'rewrite' (full redo).

    Uses the supervisor's fast model for quick classification.
    Also updates hil_summary: compresses old summary + new feedback into one string.
    """
    llm = get_node_llm("supervisor")
    structured_llm = llm.with_structured_output(FeedbackClassification)

    feedback = state.get("hil_feedback", "")
    draft = state.get("draft", "")
    old_summary = state.get("hil_summary", "")

    # ── Step 1: Classify feedback ──
    classify_prompt = (
        f"学生对以下学习计划提出了修改
       意见。\n\n"
        f"## 当前计划（前500字）\n{draft[:500]}\n\n"
        f"## 学生反馈\n{feedback}\n\n"
        f"判断这个反馈需要的修改程度：\n"
        f"- tweak: 只需要局部微调（如调整某天科目、修改时间、增删某个小项）\n"
        f"- rewrite: 需要重新规划（如整体思路不对、完全不符合需求、需要换方向）"
    )

    try:
        result = await structured_llm.ainvoke([
            SystemMessage(content="你是一个学习计划修改分类器。根据学生反馈判断需要微调还是重写。"),
            HumanMessage(content=classify_prompt),
        ])
        route = result.route
    except Exception:
        logger.warning("Feedback classification failed, defaulting to tweak")
        route = "tweak"

    # ── Step 2: Compress summary (overwrite, not append) ──
    # Simple concatenation with truncation — keeps it bounded
    if old_summary:
        new_summary = f"历史修改摘要: {old_summary[:200]}\n最新反馈: {feedback[:500]}"
    else:
        new_summary = f"用户反馈: {feedback[:500]}"

    if route == "rewrite":
        # REWRITE: Clear adversarial state, treat feedback as fresh direction
        return {
            "feedback_route": "rewrite",
            "hil_summary": new_summary,
            "revision_notes": feedback,  # drafter uses this as primary input
            # Reset adversarial loop state — start fresh
            "adv_round": 0,
            "draft": "",
            "academic_verdict": "",
            "academic_reason": "",
            "emotional_verdict": "",
            "emotional_reason": "",
            "consensus": False,
        }
    else:
        # TWEAK: Keep draft intact, pass feedback to plan_tweak
        return {
            "feedback_route": "tweak",
            "hil_summary": new_summary,
        }
```

#### Node: `plan_tweak_node` — Surgical edit of existing draft

```python
@traced_node
async def plan_tweak_node(state: TutorState) -> dict[str, Any]:
    """Make a targeted edit to the plan based on user feedback.

    Single LLM call — no reviewer loop needed for minor edits.
    """
    llm = get_node_llm("planner")
    temperature = get_setting("planner.temperature", 0.7)
    fallback = get_fallback_llm(temperature=temperature)

    draft = state.get("draft", "")
    feedback = state.get("hil_feedback", "")
    summary = state.get("hil_summary", "")

    prompt = (
        f"请根据学生的反馈对以下学习计划进行**局部微调**。\n"
        f"只修改学生提到的部分，保持其他内容不变。\n\n"
        f"## 当前计划\n{draft}\n\n"
        f"## 学生反馈\n{feedback}\n\n"
    )
    if summary:
        prompt += f"## 修改历史摘要\n{summary}\n\n"
    prompt += "请输出修改后的完整计划："

    messages = [
        SystemMessage(content=load_prompt("plan_drafter_system")),
        HumanMessage(content=prompt),
    ]

    with traced_llm_call(
        model_name=os.getenv("DEEPSEEK_MODEL", "deepseek-chat"),
        node_name="plan_tweak",
        temperature=temperature,
    ) as span:
        response = await async_invoke_with_fallback(
            llm, messages, fallback=fallback, span=span,
        )

    return {"draft": response.content}
```

### Task FE-ENH-02: Wire Feedback into page.tsx

#### 1. Add `handleFeedback` callback:
```typescript
const handleFeedback = useCallback(async (feedback: string) => {
    const threadId = threadIdRef.current
    if (!threadId) {
      setLogs((prev) => [
        ...prev,
        { type: "error", message: "[ERROR] No thread_id — cannot send feedback", ts: timestamp() },
      ])
      return
    }

    setIsResuming(true)
    setLogs((prev) => [
      ...prev,
      { type: "info", message: `[INFO] Sending feedback: ${feedback.slice(0, 40)}...`, ts: timestamp() },
    ])

    try {
      const body = await fetchWithErrorHandling(`${API_BASE_URL}/resume`, {
        method: "POST",
        headers: { "Content-Type": "application/json", ...getAuthHeaders() },
        body: JSON.stringify({ thread_id: threadId, feedback }),
      })

      if (!body) return

      // Hide PlanReview while system processes feedback
      setIsInterrupted(false)
      setInterruptDraft("")

      // Create new assistant message placeholder for the revised plan streaming
      const newAsstId = (Date.now() + 1).toString()
      assistantMessageIdRef.current = newAsstId
      setMessages((prev) => [
        ...prev,
        { id: newAsstId, role: "assistant", content: "" },
      ])

      await consumeSSEStream(body)

      setLogs((prev) => [
        ...prev,
        { type: "info", message: "[INFO] Feedback revision complete.", ts: timestamp() },
      ])
    } catch (error: any) {
      setLogs((prev) => [
        ...prev,
        { type: "error", message: `[ERROR] Feedback failed: ${error.message}`, ts: timestamp() },
      ])
    } finally {
      setIsResuming(false)
      setIsLoading(false)
    }
  }, [fetchWithErrorHandling, consumeSSEStream])
```

**Why `handleFeedback` creates a NEW assistant message**: The revised plan is new content that streams through `drafter` or `plan_tweak` nodes. Creating a new message placeholder ensures the streamed tokens don't overwrite the previous plan text in the same message bubble.

#### 2. Pass `onFeedback` to PlanReview:

1. **User**: "帮我制定一个高三数学复习计划"
2. **System**: Supervisor classifies → `planning` → search_policy → gather_intel → drafter → reviewers → consensus → `plan_output`
3. **plan_output**: calls `interrupt(draft)` → graph PAUSES
4. **SSE**: `{"type":"interrupt","draft":"## 每日复习计划\n...","thread_id":"abc123"}`
5. **Frontend**: Shows `PlanReview` with draft in textarea + empty feedback input

6. **User types feedback**: "把周三的数学改成物理，另外加一个每周测验"
7. **User clicks**: "要求修改"
8. **Frontend**: `POST /resume` with `{"thread_id":"abc123","feedback":"把周三的数学改成物理..."}`
9. **Backend**: `Command(resume={"action":"feedback","text":"把周三的数学改成物理..."})`
10. **plan_output_node**: receives dict → sets `hil_action="feedback"`, `hil_feedback="把周三的数学改成物理..."`
11. **feedback_router**: classifies as "tweak" (局部微调), updates `hil_summary`
12. **plan_tweak**: LLM edits draft in-place, returns modified `draft`
13. **plan_output**: calls `interrupt(new_draft)` → graph PAUSES again
14. **SSE**: `{"type":"interrupt","draft":"## 每日复习计划\n...(updated)...","thread_id":"abc123"}`
15. **Frontend**: `PlanReview` re-renders with new draft, feedback input cleared

16. **User types feedback**: "整个计划太松了，我需要更紧凑的安排，每天至少学8小时"
17. **User clicks**: "要求修改"
18. **Frontend**: `POST /resume` with `{"thread_id":"abc123","feedback":"整个计划太松了..."}`
19. **feedback_router**: classifies as "rewrite" (需要重新规划), clears adversarial state, updates `hil_summary`
20. **drafter**: starts fresh with `revision_notes="整个计划太松了..."` and `hil_summary` context
21. **Full adversarial loop**: drafter → reviewers → consensus → plan_output
22. **plan_output**: `interrupt(completely_new_draft)` → PAUSES

24. **User**: Reviews the new plan, satisfied

29. **SSE**: `{"type":"done"}`
30. **Frontend**: Shows final plan in chat, `PlanReview` hidden

## 10. Design Decisions & Rationale

### Why single `hil_summary` instead of a list of feedback rounds?
- Each feedback round adds ~200-500 chars of raw text
- After 5 rounds, that's 1000-2500 chars of accumulated context being sent to every LLM call
- Single summary caps at ~700 chars (200 old + 500 new), bounded regardless of round count
- The drafter/tweak LLM only needs to know the current direction, not the full history

### Why Feedback Router instead of always tweaking or always rewriting?
- "把时间从下午改到晚上" → tweak (1 LLM call, fast)
- "这个计划完全不对，我需要文科复习计划不是理科" → rewrite (full adversarial loop, thorough)
- User experience: small changes feel instant, big changes get proper attention
- Cost: tweaks skip 3-5 LLM calls (two reviewers + consensus + potential revision)

### Why `plan_tweak` skips the reviewer loop?
- Minor edits (swap one subject, adjust one time slot) don't need academic/emotional review
- The original plan already passed both reviewers before reaching the user
- Keeps tweak path fast (~2s) vs rewrite path (~8-15s)

### Why `plan_tweak` is in `ALLOWED_NODES` (streaming)?
- The tweak node generates a complete modified plan via LLM
- Streaming the output gives the user real-time feedback that "something is happening"
- Without streaming, the user would see a blank screen for 2-3 seconds during the tweak
